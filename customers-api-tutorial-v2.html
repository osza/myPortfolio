This tutorial demonstrates a simple end‑to‑end workflow for managing customer data with a Customers API. The API, URLs, and example data are fictional and are provided only to illustrate structure, flow, and writing style.

By the end of this tutorial, you will have a working CRUD workflow for a customer resource and a reusable pattern you can apply to other endpoints in a typical SaaS product.

The examples focus on developer experience, clarity, and consistency—showing how documentation can help engineers ship features faster while giving product teams and stakeholders a reliable reference.

Before you start

Requirements and setup

Before you begin, make sure you have:

  • An API access token with permission to call the Customers API.
  • The base URL for your environment (for example, sandbox or production).
  • A tool for sending HTTP requests, such as curl or Postman.
  • Network access to the /api/1.0/customers endpoint.

In the examples below, replace {base_url} and {access_token} with your own values. In a real application, you typically store these values in environment variables such as BASE_URL and ACCESS_TOKEN.

For safety, use a dedicated sandbox environment and a non‑production token while testing this workflow.

Customers API snapshot

Endpoint overview

The Customers API exposes a RESTful resource for managing customer data. This tutorial uses the following endpoints:

  • POST /api/1.0/customers — Create a new customer.
  • GET /api/1.0/customers/{id} — Retrieve a customer by ID.
  • PATCH /api/1.0/customers/{id} — Update selected fields on a customer.
  • DELETE /api/1.0/customers/{id} — Delete a customer.

Each endpoint uses bearer token authentication and returns JSON responses with standard HTTP status codes.

1. Create a customer

POST /api/1.0/customers · 201 Created

Send a POST request to create a new customer resource. On success, the API returns a stable id that you will reuse in later requests. 

curl -X POST "{base_url}/api/1.0/customers" \
  -H "Authorization: Bearer {access_token}" \
  -H "Content-Type: application/json" \
  -d '{
    "full_name": "John Doe",
    "email": "john@doe.com"
  }'
async function createCustomer() {
  const response = await fetch(`${baseUrl}/api/1.0/customers`, {
    method: "POST",
    headers: {
      Authorization: `Bearer ${accessToken}`,
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      full_name: "John Doe",
      email: "john@doe.com"
    })
  });

  if (!response.ok) {
    // In a real app, handle 4xx/5xx responses appropriately.
    throw new Error(`Request failed with status ${response.status}`);
  }

  const customer = await response.json();
  return customer;
}

2. Retrieve a customer

GET /api/1.0/customers/{id} · 200 OK

Send a GET request to retrieve the customer resource by using the id returned in the previous step. 

curl -X GET "{base_url}/api/1.0/customers/9c7c97a8-ac45-bh78-8b38-26a74fdf1fgh" \
  -H "Authorization: Bearer {access_token}"
async function getCustomer(id) {
  const response = await fetch(
    `${baseUrl}/api/1.0/customers/${id}`,
    {
      method: "GET",
      headers: {
        Authorization: `Bearer ${accessToken}`
      }
    }
  );

  if (!response.ok) {
    throw new Error(`Request failed with status ${response.status}`);
  }

  const customer = await response.json();
  return customer;
}

Example 200 response:

{

  "full_name": "John Doe",

3. Update a customer

PATCH /api/1.0/customers/{id} · 200 OK

Send a PATCH request to update selected fields on the customer resource. In this example, you change the customer's name and add a business name. 

curl -X PATCH "{base_url}/api/1.0/customers/9c7c97a8-ac45-bh78-8b38-26a74fdf1fgh" \
  -H "Authorization: Bearer {access_token}" \
  -H "Content-Type: application/json" \
  -d '{
    "full_name": "John Smith",
    "business_name": "Blahblah Ltd"
  }'
async function updateCustomer(id) {
  const response = await fetch(
    `${baseUrl}/api/1.0/customers/${id}`,
    {
      method: "PATCH",
      headers: {
        Authorization: `Bearer ${accessToken}`,
        "Content-Type": "application/json"
      },
      body: JSON.stringify({
        full_name: "John Smith",
        business_name: "Blahblah Ltd"
      })
    }
  );

  if (!response.ok) {
    throw new Error(`Request failed with status ${response.status}`);
  }

  const customer = await response.json();
  return customer;
}

Example 200 response:

{
  "id": "9c7c97a8-ac45-bh78-8b38-26a74fdf1fgh",
  "full_name": "John Smith",
  "business_name": "Blahblah Ltd",
  "email": "john@doe.com",
  "created_at": "2022-06-13T12:03:39.979397Z",
  "updated_at": "2022-06-13T12:10:13.979397Z"
}

Only the fields included in the request body are updated. Other fields, such as email, remain unchanged.

4. Delete a customer

DELETE /api/1.0/customers/{id} · 204 No Content

Send a DELETE request to remove the customer resource permanently. 

curl -X DELETE "{base_url}/api/1.0/customers/9c7c97a8-ac45-bh78-8b38-26a74fdf1fgh" \
  -H "Authorization: Bearer {access_token}"
async function deleteCustomer(id) {
  const response = await fetch(
    `${baseUrl}/api/1.0/customers/${id}`,
    {
      method: "DELETE",
      headers: {
        Authorization: `Bearer ${accessToken}`
      }
    }
  );

  if (!response.ok) {
    throw new Error(`Request failed with status ${response.status}`);
  }

  // A 204 No Content response indicates that the customer was deleted successfully.
  return;
}

204 No Content

Error handling

Typical 4xx response

When a request is invalid, the API returns a 4xx status code and a JSON error payload describing what went wrong. This helps developers diagnose issues quickly.

Example 400 response when a required field is missing:

{
  "error": {
    "code": "invalid_request",
    "message": "The 'email' field is required.",
    "details": [
      {
        "field": "email",
        "issue": "missing"
      }
    ]
  }
}

Implementation notes and documentation patterns

How this sample is structured

  • Use the same customer id throughout the workflow.
  • Include the Authorization header in every request.
  • For requests that include a JSON body, such as POST or PATCH, include the Content-Type: application/json header.
  • Use PATCH when you want to update selected fields instead of replacing the entire object.
  • The tutorial combines narrative steps, multi‑language code snippets, and a concise API snapshot, illustrating a documentation pattern that scales across endpoints.
