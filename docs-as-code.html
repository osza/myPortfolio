How I work with docs-as-code

I treat documentation as part of the product: versioned, reviewable, and deployable from the same workflows as the code. This page describes how I like to design and run a docs-as-code workflow for API and integration documentation.

← Back to main portfolio

Blue-sky vision

In an ideal setup, all user-facing API and integration documentation lives in a single, version-controlled repository together with the specifications and example payloads.

API reference is generated from a source of truth specification (typically OpenAPI in YAML), while tutorials and integration guides are written in Markdown with structured front matter for metadata, navigation, and search.

The documentation site is built automatically from this repository using a static site generator or API documentation tool, and deployed on every successful build of the main branch. Each product or API change triggers a documentation task, so that code and docs can be reviewed and released together.

Working from OpenAPI as source of truth

For API documentation, I prefer to treat the OpenAPI specification as the main source of truth. The spec in YAML defines endpoints, request and response bodies, status codes, and reusable components.

From there, I use tools that generate the core reference pages and interactive explorers, and then layer hand-written explanations, examples, and integration guides on top. This keeps the mechanical parts of the documentation aligned with the actual API surface, while still allowing room for narrative content.

  • OpenAPI YAML in the repository as a versioned artefact
  • Generated or semi-generated API reference from the spec
  • Markdown guides that explain use cases, workflows, and edge cases
  • Shared snippets for authentication, pagination, and error handling

Day-to-day workflow

On a daily basis, I work from a docs repository that mirrors the product structure: API specs, reference, guides, and shared snippets live next to each other.

For documentation work, I create focused branches such as feature/new-endpoint or fix/error-responses, make small commits, and open pull requests for review by engineers and product owners.

When the API changes, I start by updating the OpenAPI YAML, then regenerate or adjust the reference content, and finally review any tutorials or integration guides that depend on the changed behaviour.

Before merging, I run automated checks (build, link validation, linting) so that broken links or structural issues are caught early, not on the live site.

Collaboration and reviews

I treat documentation changes as a shared responsibility between product, engineering, and documentation. For new features, I usually start from a short outline or doc plan, review it with the product manager and lead engineer, and only then move to full drafts in Markdown.

Pull requests are the main review channel. Engineers focus on technical accuracy and behaviour, while product and support teams review coverage, terminology, and customer scenarios. This gives each merged change a visible review history and a clear rationale.

Once changes are approved and merged, the same pipeline that builds and deploys the docs also leaves a traceable record in release notes, so it is easy to understand when and why documentation for a feature was updated.

Tooling and automation

I rely on Git for version control and keep documentation in the same ecosystem as the code, using branches, pull requests, and tags or releases to snapshot meaningful changes.

For API specifications, I work with OpenAPI in YAML or JSON and use tools that can validate the spec and generate parts of the reference. For narrative content, I prefer Markdown with simple front matter fields for titles, descriptions, and navigation.

Continuous integration runs builds and checks on every change to the main branch. Over time, I like to add automated checks for style, links, and structure so that documentation remains maintainable as the product grows.

This portfolio itself is versioned in Git and released through GitHub releases, as a simple example of treating documentation as an evolving, versioned product.

In practice, I adapt this flow to the tools and constraints of each team, but the core idea stays the same: one source of truth for the API, docs stored as code, and an automated path from change to published documentation.
