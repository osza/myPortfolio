Story: Giving players real ownership

Narrative context · Web2 vs Web3

Imagine you are building an action RPG where players collect weapons, skins, and artifacts. In a traditional Web2 stack, each item is a row in a database table, fully controlled by the game studio.

On Sui, each item becomes an on‑chain object: a non‑fungible token (NFT) or a custom asset governed by smart contracts written in Move. Players own these assets in their wallets, can trade them on external marketplaces, and can carry them into other experiences. [web:34][web:18][web:20]

This tutorial shows how a game backend exposes a familiar REST‑style API to the game client, while executing JSON‑RPC calls to Sui under the hood to mint, update, transfer, and burn assets on the blockchain. [web:10][web:24][web:31]

Architecture at a glance

Game backend · Sui node · Player wallet

The game client never talks directly to the blockchain. Instead, it uses a game API that you control. That API in turn uses a Sui RPC provider (for example QuickNode or Alchemy) to send JSON‑RPC requests to the network. [web:10][web:35][web:24]

At a high level:

  • The game client calls your REST API, for example POST /api/game/v1/assets.
  • Your backend verifies the request, builds a Sui transaction, and sends it via JSON‑RPC.
  • The player signs the transaction in their wallet, and Sui finalizes it on‑chain.
  • Your backend stores lightweight metadata and mirrors on‑chain state for fast queries.

This pattern lets you keep a clean, product‑oriented API surface for the game team, while still benefiting from Sui’s object‑centric, high‑throughput design. [web:18][web:21][web:42]

Before you start

Requirements: wallet, endpoint, smart contracts

To follow this tutorial end‑to‑end, you need:

  • A Sui wallet on testnet with some SUI for gas (for example the official Sui Wallet).
  • A Sui RPC endpoint from a provider such as QuickNode or Alchemy.
  • A basic game backend that can send HTTP requests and handle JSON.
  • A Move package deployed to Sui that defines your game asset type (for example a weapon NFT).

In the examples below, replace {sui_endpoint} and {api_key} with your own values from your RPC provider dashboard, and {package_id} with the ID of your published Move package. [web:10][web:35][web:36][web:37]

1. Mint a weapon for a new player

POST /api/game/v1/assets · JSON‑RPC mint

When a new player finishes the tutorial level, you want to give them their first weapon as an on‑chain asset. The game client calls your API, and your backend mints a new object on Sui. 

curl -X POST "{base_url}/api/game/v1/assets" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {player_session_token}" \
  -d '{
    "type": "weapon",
    "name": "Blade of Dawn",
    "rarity": "epic"
  }'
async function mintWeaponForPlayer({ playerAddress, name, rarity }) {
  const payload = {
    jsonrpc: "2.0",
    id: 1,
    method: "sui_moveCall",
    params: [
      "{package_id}",          // Your Move package ID
      "game_assets",           // Module name
      "mint_weapon",           // Function name
      [name, rarity],          // Function arguments
      [playerAddress],         // Recipient address
      { maxGasBudget: 10_000 } // Gas budget
    ]
  };

  const response = await fetch("{sui_endpoint}", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer {api_key}`
    },
    body: JSON.stringify(payload)
  });

  if (!response.ok) {
    throw new Error(`Sui RPC failed with status ${response.status}`);
  }

  const result = await response.json();
  return result;
}

On success, the Sui RPC response includes the ID of the newly minted weapon object. Your backend can persist this ID alongside game metadata and return it to the client. [web:10][web:35][web:41]

2. Upgrade a weapon after a boss fight

PATCH /api/game/v1/assets/{object_id} · JSON‑RPC update

After a successful boss fight, you might increase the weapon’s power level and change its rarity. In a Web2 backend, this is a simple update in a database row. On Sui, you modify the on‑chain object via a Move call. [web:18][web:20][web:26] 

curl -X PATCH "{base_url}/api/game/v1/assets/{object_id}" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {player_session_token}" \
  -d '{
    "power_level": 42,
    "rarity": "legendary"
  }'
async function upgradeWeapon({ objectId, powerLevel, rarity }) {
  const payload = {
    jsonrpc: "2.0",
    id: 2,
    method: "sui_moveCall",
    params: [
      "{package_id}",
      "game_assets",
      "upgrade_weapon",
      [objectId, powerLevel, rarity],
      [], // No new recipients; weapon stays with current owner
      { maxGasBudget: 8_000 }
    ]
  };

  const response = await fetch("{sui_endpoint}", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer {api_key}`
    },
    body: JSON.stringify(payload)
  });

  if (!response.ok) {
    throw new Error(`Sui RPC failed with status ${response.status}`);
  }

  const result = await response.json();
  return result;
}

This step demonstrates Sui’s resource‑oriented model: the weapon is a single, non‑duplicable object; you change its state over time instead of cloning or copying it. [web:18][web:26][web:28]

3. Trade a weapon between players

POST /api/game/v1/trades · JSON‑RPC transfer

One of the main reasons to use web3 in games is open economies. A player can list a weapon on a marketplace, another player can buy it, and your game can immediately reflect the new ownership. [web:34][web:11][web:25] 

curl -X POST "{base_url}/api/game/v1/trades" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {seller_session_token}" \
  -d '{
    "object_id": "{weapon_object_id}",
    "buyer_address": "{buyer_sui_address}",
    "price_token": "SUI",
    "price_amount": "10"
  }'
async function transferWeapon({ objectId, fromAddress, toAddress }) {
  const payload = {
    jsonrpc: "2.0",
    id: 3,
    method: "sui_transferObject",
    params: [
      fromAddress,
      objectId,
      toAddress,
      { maxGasBudget: 5_000 }
    ]
  };

  const response = await fetch("{sui_endpoint}", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer {api_key}`
    },
    body: JSON.stringify(payload)
  });

  if (!response.ok) {
    throw new Error(`Sui RPC failed with status ${response.status}`);
  }

  const result = await response.json();
  return result;
}

Unlike a SaaS backend, you are not just updating an owner_id column. Ownership changes on‑chain and is visible to any application that queries the Sui network. [web:21][web:34][web:41]

4. Retire a weapon from the economy

DELETE‑like workflow · JSON‑RPC burn

To keep your economy sustainable, you might remove some assets from circulation—for example when a weapon breaks permanently in a story event, or when a limited‑time event ends. [web:34][web:22][web:25]

In your API, this looks like a delete‑style operation; in your Move code, you burn or otherwise retire the object so it cannot be used or traded anymore. [web:18][web:20][web:26]

Error handling and finality

Gas, pending states, RPC failures

Working with a blockchain introduces new failure modes compared to a classic SaaS API: insufficient gas, invalid signatures, or network congestion. [web:35][web:42][web:40]

Your tutorial should explain:

  • How to report pending vs finalized transaction status to the game client.
  • How to surface RPC errors with actionable messages for developers.
  • How to retry or reconcile state when on‑chain data and local cache diverge.
