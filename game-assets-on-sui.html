Story: Giving players real ownership

Narrative context · Web2 vs Web3

Imagine you are building an action RPG where players collect weapons, skins, and artifacts. In a traditional Web2 stack, each item is a row in a database table, fully controlled by the game studio.

On Sui, each item becomes an on‑chain object: a non‑fungible token (NFT) or a custom asset governed by smart contracts written in Move. Players own these assets in their wallets, can trade them on external marketplaces, and can carry them into other experiences.

This tutorial shows how a game backend exposes a familiar REST‑style API to the game client, while Move modules and RPC clients in Rust or C++ interact with Sui under the hood to mint, update, transfer, and retire assets on the blockchain.

The examples in this tutorial are fictional and simplified for clarity, but they follow the real architectural patterns recommended in Sui gaming and Move documentation, including object‑centric assets, programmable transactions, and Rust SDK integration.

Architecture at a glance

Game engine · Backend service · Sui node

The game client never talks directly to the blockchain. Instead, it uses a game API that you control. That API in turn uses a Sui RPC provider (for example QuickNode or Alchemy) or an SDK to send requests to the network.

At a high level:

  • The game client calls your REST API, for example POST /api/game/v1/assets.
  • Your backend (for example in Rust) verifies the request and builds a Sui transaction.
  • A JSON‑RPC client (Rust or C++) sends the transaction to a Sui node.
  • The player signs the transaction in their wallet, and Sui finalizes it on‑chain.
  • Your backend stores lightweight metadata and mirrors on‑chain state for fast queries.

This pattern lets you keep a clean, product‑oriented API surface for the game team, while still benefiting from Sui’s object‑centric, high‑throughput design.

Before you start

Requirements: wallet, endpoint, smart contracts

To follow this tutorial end‑to‑end, you need:

  • A Sui wallet on testnet with some SUI for gas (for example the official Sui Wallet).
  • A Sui RPC endpoint from a provider such as QuickNode or Alchemy.
  • A backend service that can send HTTP requests and handle JSON (for example in Rust).
  • A Move package deployed to Sui that defines your game asset type (for example a weapon NFT).

In the examples below, replace {sui_endpoint} and {api_key} with your own values from your RPC provider dashboard, and {package_id} with the ID of your published Move package.

Move: Define the weapon asset

Smart contract logic · Resource‑oriented model

On Sui, game assets are defined as Move structs with resource semantics: they cannot be duplicated or accidentally lost, and ownership is explicit. The following simplified example shows a weapon object and an entry function that mints it to the player. 

module game_assets::weapon {
  use sui::object::{Self, UID};
  use sui::tx_context::{Self, TxContext};
  use sui::transfer;

  struct Weapon has key, store {
    id: UID,
    name: vector<u8>,
    power_level: u64,
    rarity: vector<u8>
  }

  public entry fun mint_weapon(
    name: vector<u8>,
    power_level: u64,
    rarity: vector<u8>,
    ctx: &mut TxContext
  ) {
    let weapon = Weapon {
      id: object::new(ctx),
      name,
      power_level,
      rarity
    };
    transfer::public_transfer(weapon, tx_context::sender(ctx));
  }
}

In a production game, you would add more fields (for example durability or element type) and additional entry functions for upgrading, trading, or burning weapons.

1. Mint a weapon for a new player

POST /api/game/v1/assets · Sui RPC mint

When a new player finishes the tutorial level, you want to give them their first weapon as an on‑chain asset. The game client calls your API, and your backend mints a new object on Sui using the Move function defined above. 

curl -X POST "{base_url}/api/game/v1/assets" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {player_session_token}" \
  -d '{
    "type": "weapon",
    "name": "Blade of Dawn",
    "rarity": "epic"
  }'
use sui_sdk::SuiClientBuilder;
use sui_sdk::types::base_types::SuiAddress;

async fn mint_weapon_for_player(
    client: &sui_sdk::SuiClient,
    player: SuiAddress,
    name: String,
    rarity: String,
) -> anyhow::Result<()> {
    // High-level sketch; production code would use
    // programmable transactions and strongly typed arguments.
    let tx = client.transaction_builder()
        .move_call(
            "{package_id}".parse()?,   // Move package ID
            "game_assets".to_string(), // Module
            "mint_weapon".to_string(), // Function
            vec![name.into_bytes(), rarity.into_bytes()],
            Some(player),
        )
        .await?;

    let response = client.execute_transaction(tx).await?;
    println!("Minted weapon tx: {:?}", response);
    Ok(())
}
// Simplified C++ example sending JSON-RPC to a Sui endpoint
json payload = {
  {"jsonrpc", "2.0"},
  {"id", 1},
  {"method", "sui_moveCall"},
  {"params", json::array({
      "{package_id}",           // Move package ID
      "game_assets",            // Module
      "mint_weapon",            // Function
      json::array({ name, rarity }),
      json::array({ playerAddress }),
      json::object({ { "maxGasBudget", 10000 } })
  })}
};

auto response = httpClient.post("{sui_endpoint}", payload.dump());

if (!response.ok()) {
  throw std::runtime_error("Sui RPC failed");
}

// Parse JSON, extract weapon object ID, store it in game state.

On success, the Sui RPC response includes the ID of the newly minted weapon object. Your backend or engine can persist this ID alongside game metadata and return it to the client.

2. Upgrade a weapon after a boss fight

PATCH /api/game/v1/assets/{object_id} · Sui RPC update

After a successful boss fight, you might increase the weapon’s power level and change its rarity. In a Web2 backend, this is a simple update in a database row. On Sui, you modify the on‑chain object via a Move call. 

curl -X PATCH "{base_url}/api/game/v1/assets/{object_id}" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {player_session_token}" \
  -d '{
    "power_level": 42,
    "rarity": "legendary"
  }'
module game_assets::weapon_upgrade {
  use sui::object::{Self, UID};
  use sui::tx_context::{Self, TxContext};

  // Simplified example: upgrade an existing weapon
  public entry fun upgrade_weapon(
    weapon: &mut game_assets::weapon::Weapon,
    new_power_level: u64,
    new_rarity: vector<u8>
  ) {
    weapon.power_level = new_power_level;
    weapon.rarity = new_rarity;
  }
}

This step demonstrates Sui’s resource‑oriented model: the weapon is a single, non‑duplicable object; you change its state over time instead of cloning or copying it.

3. Trade a weapon between players

POST /api/game/v1/trades · Sui RPC transfer

One of the main reasons to use web3 in games is open economies. A player can list a weapon on a marketplace, another player can buy it, and your game can immediately reflect the new ownership. 

curl -X POST "{base_url}/api/game/v1/trades" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {seller_session_token}" \
  -d '{
    "object_id": "{weapon_object_id}",
    "buyer_address": "{buyer_sui_address}",
    "price_token": "SUI",
    "price_amount": "10"
  }'
// Simplified C++ transfer using JSON-RPC
json payload = {
  {"jsonrpc", "2.0"},
  {"id", 2},
  {"method", "sui_transferObject"},
  {"params", json::array({
      sellerAddress,
      weaponObjectId,
      buyerAddress,
      json::object({ { "maxGasBudget", 5000 } })
  })}
};

auto response = httpClient.post("{sui_endpoint}", payload.dump());
if (!response.ok()) {
  throw std::runtime_error("Sui transfer failed");
}

// Ownership now changed on-chain; game can query Sui to confirm.

Unlike a SaaS backend, you are not just updating an owner_id column. Ownership changes on‑chain and is visible to any application that queries the Sui network.

4. Retire a weapon from the economy

Delete‑style workflow · Burn logic in Move

To keep your economy sustainable, you might remove some assets from circulation—for example when a weapon breaks permanently in a story event, or when a limited‑time event ends.

In your API, this looks like a delete‑style operation; in your Move code, you burn or otherwise retire the object so it cannot be used or traded anymore. 

module game_assets::weapon_burn {
  use sui::object;
  use game_assets::weapon::Weapon;

  public entry fun burn_weapon(weapon: Weapon) {
    // Dropping the Weapon consumes the resource; depending on your
    // design, you may log events or emit metrics here.
    object::delete(weapon.id);
  }
}

Error handling and finality

Gas, pending states, RPC failures

Working with a blockchain introduces new failure modes compared to a classic SaaS API: insufficient gas, invalid signatures, or network congestion.

Your tutorial should explain:

  • How to report pending vs finalized transaction status to the game client.
  • How to surface RPC errors with actionable messages for developers.
  • How to retry or reconcile state when on‑chain data and local cache diverge.

Result

You have minted a weapon for a new player, upgraded its stats, traded it between players, and retired it from the game economy, all while keeping a clean REST‑style API surface for your game.

This pattern combines familiar Web2 practices with Sui’s object‑centric blockchain model, giving players real ownership and giving developers a scalable way to manage game assets on‑chain.
